# A conversational route that bypasses retrieval and the supervisor

Not every message a practitioner sends is a **Legal Research Query**. Greetings, self-introductions ("my name is Shameel"), thanks, small talk, and meta questions ("what can you do?") used to be forced through the single legal pipeline (`router → contextualize → retriever → synthesiser → citation_validator → grounding_check → supervisor`). The router could only pick a legal type, so a greeting was force-fit into a legal bucket, retrieved irrelevant statute chunks, and the synthesiser emitted the robotic *"There is no provision in the retrieved statute sections..."* template — and the supervisor, which requires a citation plus a disclaimer on every response, made a warm reply structurally impossible.

We add a `conversational` query type and a dedicated `conversational` node. When the router classifies a message as `conversational`, the graph routes `router → conversational → record_turn`, skipping retrieval, synthesis, and all four supervisor checks. The node generates a short, warm reply with a small LLM run at `temperature=0.7` (so repeated greetings vary in wording), mirrors the user's language, reads trimmed **Conversation History** (so "what's my name?" recalls a name given earlier), and fails closed to a static greeting constant on any error. It returns `final_response`/`draft_response` with empty `citations` and no disclaimer.

The supervisor stays strict for legal turns — this decision does **not** relax the citation/disclaimer/no-advice rules. The no-result humanization that strict mode would otherwise block is redirected into warmer `FINAL_FAILURE_RESPONSE` and `ESCALATION_RESPONSE` text.

## Considered Options

- **Relax the supervisor so legal answers can be warm without a citation.** Rejected. The citation-per-claim and disclaimer rules are the safety contract; loosening them to humanize a no-result case would just manufacture uncited "answers" or push more turns to the fallback. The warmth a user actually hits on a dead-end belongs in the canned fallback text, not in a weaker rule.
- **Symmetric router tie-break (treat ambiguous turns evenly).** Rejected. The two misroutings are not equally costly. Misrouting a greeting → legal is the bug we're fixing (robotic template, junk retrieval). Misrouting a legal query → conversational only yields a safe deflection ("I research legislation — ask me about an Act..."), which is recoverable. So the router leans legal on ties: `conversational` is reserved for *unambiguous* social/meta turns, and "when in doubt, classify as a legal type."
- **Add the route but keep every node at `temperature=0`.** Rejected. A "warm" reply at temperature 0 is byte-identical every time, so a user who says "hi" twice gets the same sentence — the opposite of human. `make_llm` gains a `temperature` parameter (default 0); only the conversational node opts into heat, so the legal nodes stay reproducible and the evals are untouched.

## Consequences

- The conversational route is **outside** the safety layer by design. This is safe only because the router's escalation keyword pre-check runs first and unchanged ("hi, am I liable?" still escalates), and because the route is reserved for unambiguous non-legal turns. Do not widen `conversational` to cover anything with legal substance — that would route real legal questions around the supervisor.
- The regression gate for this change is `python evals/run_evals.py --smoke` (10 cases incl. 2 escalation), confirming no legal case drifts to `conversational` and escalation still fires — not the full 50-case suite.
- Known limitation (accepted under strict mode): a legal query that retrieves *nothing* relevant still must satisfy the citation rule, so it either cites the nearest consulted section or falls to the (now warmer) `FINAL_FAILURE_RESPONSE`. This is an edge case — retrieval returns top-k for any real legal query — and is a candidate follow-up if it annoys during the pilot.
